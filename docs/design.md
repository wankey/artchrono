## Problem Statement

中国艺术培训市场（钢琴、舞蹈、美术、声乐等）极度分散，绝大多数小机构和独立老师在用"微信群 + Excel + 便签纸"的组合拳硬撑日常教务。核心痛点：

1. **排课**：每个学生有固定的每周上课时间（如"学生A 每周六 9-10点"），需要老师手动记忆
2. **考勤**：每次课后老师手动记录"今天谁来了"，容易漏记、错记
3. **付费周期**：常见模式是"先付 N 节课的学费"（N 因家长而异：8/10/12/20/月付/学期付），上完 N 节需要提醒家长续费
4. **续费提醒**：老师要在脑子里或纸上跟踪每个学生的"剩余课时"，< 阈值时主动联系家长，否则容易漏掉

目标用户规模：单个艺术老师 + 10-30 个学生。

MVP 落地后的核心"哇"点（V1 范围内）：

### 余额数学（按报名 / enrollment 计算）

V1 在 service 层实现，V2 改为 DB triggers：

- **付款**：`INSERT INTO payments (...)` → `enrollments.classes_remaining += payments.classes_paid`
  - 付款金额自动从 `payments.amount_cents` / `payments.classes_paid` 计算，无需手动核对
  - 老师录入付款时填：节数 N + 实际收款金额（用于记账）
- **考勤（attended）**：`INSERT INTO attendance (...)` WHERE `result = 'attended'` → `enrollments.classes_remaining -= 1`
- **考勤（其他状态）**：no_show / cancelled / make_up → 余额不变
- **报名完成**：由 teacher **手动**标记 `status = 'completed'`，`completed_at = today`（不是按固定节数自动完成）
- **升级下一级**：teacher 手动创建新 enrollment 指向下一 level，原 enrollment 标 completed
- **payment 金额可独立记账**：`payments.amount_cents` 是实际收的钱（家长可能折扣/涨价/补差价），跟 `classes_paid × exam_levels.price_cents` 不必相等

注：`classes_remaining` 允许 negative（老师忘记录付款场景），UI 红色显示但不阻止扣减。

**好处**：
- 小王同时学"钢琴 3 级"和"乐理 1 级"，两个独立余额 + 独立提醒
- level 完成由 teacher 判断（学生真"达到"该级水平），不是机械的"N 节课后自动毕业"
- payment 金额灵活（家长议价、套餐价、补差价都能记）

## scheduled_classes 生成机制

**核心问题**：class_slots 是"模板"（每周六 9-10 点），scheduled_classes 是"实例"（2026-07-11 9-10 点的具体一节）。需要从模板生成实例。

**新机制（基于付款余额）**：
- 生成数量 = `enrollments.classes_remaining`（已付费但未上完的节数）
- 每次出勤（attended）扣 1 → 下次 cron 自动补齐
- 每次付款（payment insert）→ 立即触发补齐（service 层调 generation）

**触发方式**（3 个）：

1. **Payment insert 触发**（主要）：
   - 老师在 UI 录入付款后，service 层立即调 `regenerate_for_enrollment(enrollment_id)`
   - 生成 `classes_remaining` 个未来 scheduled_classes
   - 立即生效，无需等 cron

2. **Attendance insert 触发**（次要，V1.1 加）：
   - 学生出勤后 `classes_remaining -= 1`
   - 同样触发 `regenerate_for_enrollment()`，补 1 个新 scheduled_class 到未来
   - V1 靠 weekly cron 兜底（可接受 1 周延迟）

3. **GitHub Actions weekly cron（兜底）**：
   - 每周一 02:00 UTC 跑一次
   - 检查所有 active enrollment，确保每个 enrollment 都有 `classes_remaining` 个未来 scheduled_classes
   - 修复任何遗漏（如 cron 失败、attendance 没触发 regeneration 等）

**生成算法**（per enrollment）：

```
输入：enrollment_id
1. 查 enrollment：classes_remaining, class_slot (weekday, start_time, end_time)
2. 查已有未来 scheduled_classes 数量：count(*) where enrollment_id = X and scheduled_date >= today and status = 'scheduled'
3. 差值 = classes_remaining - 已生成数
4. 如果差值 > 0：生成差值个未来日期（按 class_slot.weekday 排）
5. 如果差值 < 0：删除多余的未来 scheduled_classes（避免超额预生成）
6. 节假日不跳过（老师手动改 status）
7. 寒暑假不跳过（V1.1 加 teacher_settings.vacation_periods）
```

**例子**：

```
小王 钢琴 3 级：classes_remaining = 5
class_slot: 周六 9-10
已有未来 scheduled_classes: 3 个（7/11, 7/18, 7/25）
差值 = 5 - 3 = 2 → 生成 8/1, 8/8 两个

小王 出勤 7/25：classes_remaining = 4
7/25 status = attended
service 触发 regeneration（V1 靠 weekly cron）
已有未来 = 2（8/1, 8/8）
差值 = 4 - 2 = 2 → 再生成 8/15, 8/22

家长付款 5 节：classes_remaining = 9
payment insert 触发 regeneration
已有未来 = 4
差值 = 9 - 4 = 5 → 生成 8/29, 9/5, 9/12, 9/19, 9/26
```

**schema 约束**（已有）：`UNIQUE (class_slot_id, scheduled_date)` 让 upsert 幂等。

**scripts/regenerate-for-enrollment.mjs 伪代码**（Revision 18 升级：支持多 class_slot per enrollment）：
```javascript
import { createClient } from '@supabase/supabase-js';
import 'dotenv/config';

const supabase = createClient(
  process.env.SUPABASE_URL,
  process.env.SUPABASE_SERVICE_ROLE_KEY  // ⚠️ 仅 GitHub Actions 使用，客户端绝不能 bundle
);

function findNextWeekdayDate(fromDate, weekday) {
  const d = new Date(fromDate);
  while (d.getDay() !== weekday) d.setDate(d.getDate() + 1);
  return d;
}

async function regenerateForEnrollment(enrollmentId) {
  const { data: enrollment } = await supabase
    .from('enrollments')
    .select('*')
    .eq('id', enrollmentId)
    .single();

  if (!enrollment || enrollment.status !== 'active') return;

  // 关键：枚举该 enrollment 的所有 active class_slots
  const { data: slots } = await supabase
    .from('class_slots')
    .select('*')
    .eq('enrollment_id', enrollmentId)
    .eq('active', true)
    .order('weekday');  // 排序确保轮询分配有序

  if (!slots?.length) return;

  const targetCount = enrollment.classes_remaining;
  const today = new Date().toISOString().split('T')[0];

  // 查已有未来 scheduled_classes
  const { data: existingDates } = await supabase
    .from('scheduled_classes')
    .select('scheduled_date')
    .eq('enrollment_id', enrollmentId)
    .gte('scheduled_date', today)
    .eq('status', 'scheduled')
    .order('scheduled_date');

  const taken = new Set(existingDates?.map(r => r.scheduled_date) ?? []);
  const diff = targetCount - taken.size;

  if (diff === 0) return;

  if (diff > 0) {
    // 多 slot 分配算法：轮流从各 slot 找下一个未来日期
    const slotCursors = slots.map(s => ({
      slot: s,
      nextDate: findNextWeekdayDate(today, s.weekday),
    }));
    const toInsert = [];
    let slotIdx = 0;

    while (toInsert.length < diff) {
      const cursor = slotCursors[slotIdx % slotCursors.length];
      const dateStr = cursor.nextDate.toISOString().split('T')[0];
      if (!taken.has(dateStr)) {
        toInsert.push({
          class_slot_id: cursor.slot.id,
          student_id: enrollment.student_id,
          enrollment_id: enrollmentId,
          teacher_id: enrollment.teacher_id,
          scheduled_date: dateStr,
          start_time: cursor.slot.start_time,
          end_time: cursor.slot.end_time,
          status: 'scheduled',
        });
        taken.add(dateStr);
      }
      cursor.nextDate = new Date(cursor.nextDate.getTime() + 7 * 24 * 60 * 60 * 1000);
      slotIdx++;
    }
    if (toInsert.length) await supabase.from('scheduled_classes').insert(toInsert);
  } else {
    // 删除多余的 future scheduled_classes（最新的先删）
    const excess = -diff;
    const { data: toDelete } = await supabase
      .from('scheduled_classes')
      .select('id')
      .eq('enrollment_id', enrollmentId)
      .gte('scheduled_date', today)
      .eq('status', 'scheduled')
      .order('scheduled_date', { ascending: false })
      .limit(excess);
    if (toDelete?.length) await supabase.from('scheduled_classes').delete().in('id', toDelete.map(r => r.id));
  }
}
```

**scripts/regenerate-all-enrollments.mjs 伪代码**（weekly cron 兜底用）：
```javascript
const { data: enrollments } = await supabase
  .from('enrollments')
  .select('id')
  .eq('status', 'active');

for (const e of enrollments ?? []) {
  await regenerateForEnrollment(e.id);
}
```

**🔒 服务端唯一调用方**：上述脚本**只能在 GitHub Actions 中跑**（用 service role key）。Tauri 客户端绝不能 bundle service role key。如果客户端需要 trigger 立即 regeneration，必须通过 Postgres RPC（authenticated user + RLS）。

**scripts/sync-holidays.mjs 伪代码**：
```javascript
import { createClient } from '@supabase/supabase-js';
import 'dotenv/config';

const supabase = createClient(...);

const currentYear = new Date().getFullYear();
const years = [currentYear, currentYear + 1];

for (const year of years) {
  try {
    const res = await fetch(`http://timor.tech/api/holiday/year/${year}`);
    const json = await res.json();
    const records = Object.entries(json.holiday ?? {}).map(([key, info]) => {
      const [month, day] = key.split('-');
      return {
        date: `${year}-${month}-${day}`,
        name: info.name,
        type: info.holiday ? 'holiday' : 'workday',
        source: 'timor.tech',
        fetched_at: new Date().toISOString(),
      };
    });
    if (records.length > 0) {
      await supabase.from('holidays').upsert(records, { onConflict: 'date' });
      console.log(`Synced ${records.length} holidays for ${year}`);
    }
  } catch (err) {
    console.error(`Failed to sync ${year}:`, err.message);
    // 失败不中断 —— 下次 cron 重试
  }
}
```

## RLS Policies

```sql
ALTER TABLE students ENABLE ROW LEVEL SECURITY;
ALTER TABLE courses ENABLE ROW LEVEL SECURITY;
ALTER TABLE exam_levels ENABLE ROW LEVEL SECURITY;
ALTER TABLE enrollments ENABLE ROW LEVEL SECURITY;
ALTER TABLE class_slots ENABLE ROW LEVEL SECURITY;
ALTER TABLE scheduled_classes ENABLE ROW LEVEL SECURITY;
ALTER TABLE attendance ENABLE ROW LEVEL SECURITY;
ALTER TABLE payments ENABLE ROW LEVEL SECURITY;
ALTER TABLE reminders ENABLE ROW LEVEL SECURITY;
ALTER TABLE holidays ENABLE ROW LEVEL SECURITY;

CREATE POLICY "teacher_owns_students" ON students
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_courses" ON courses
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_exam_levels" ON exam_levels
  FOR ALL TO authenticated
  USING (course_id IN (SELECT id FROM courses WHERE teacher_id = (SELECT auth.uid())))
  WITH CHECK (course_id IN (SELECT id FROM courses WHERE teacher_id = (SELECT auth.uid())));

CREATE POLICY "teacher_owns_enrollments" ON enrollments
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_class_slots" ON class_slots
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_scheduled_classes" ON scheduled_classes
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_attendance" ON attendance
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_payments" ON payments
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "teacher_owns_reminders" ON reminders
  FOR ALL TO authenticated
  USING (teacher_id = (SELECT auth.uid()))
  WITH CHECK (teacher_id = (SELECT auth.uid()));

CREATE POLICY "holidays_readable_by_all" ON holidays
  FOR SELECT TO authenticated USING (true);

-- 未来机构版：见 §"Future Org Migration Scope"
```

**注**：`exam_levels` 的 RLS 通过 join courses 表检查 teacher_id，避免在每个 exam_level 上冗余存 teacher_id 字段。所有相关表都反规范化了 teacher_id（除 exam_levels 走 join），保持 RLS 简单直接。

## Business Rules

### 1. 缺勤（no_show）处理

- 老师标记 no_show → 余额不扣减
- 学生可申请补课 → 老师新建一条 scheduled_classes，type = 'make_up'
- make_up 是独立的 scheduled_class（数据上与原 no_show 并列）
- make_up class 上 attended → 扣 1 余额；no_show → 不扣

### 2. 取消（cancelled）处理

- 老师/学生取消 → 余额不扣减
- 系统不自动生成新的 scheduled_class（按需手动）

### 3. 节假日处理

- **不自动跳过**：节假日当天也照常生成 scheduled_classes，老师自主决定是否上课
- **节假日数据仍同步**（`date-holidays` + timor.tech → `holidays` 表）：
  - 用于 UI 显示（日期旁边标注 "国庆节" 等）
  - 提醒老师（"今天有课，但也是节假日"）
  - 但**不参与生成逻辑**（scheduled_classes 一律生成）
- **老师操作**：节假日当天课程，teacher 可在 UI 一键改为 `status='cancelled'`（不扣余额）或 `status='make_up'`（补课）
- **寒暑假**：每个老师在 settings 中标记假期区间，区间内**不生成** scheduled_class（这是个明确的"长期不上课"决定，与节假日的"临时调课"性质不同）
- **数据刷新**：
  - GitHub Actions annual cron（每年 1 月 1 日 02:00 UTC）跑 `scripts/sync-holidays.mjs`，从 timor.tech 拉当年 + 次年数据，upsert 到 DB
  - App 首次启动时检测 DB 中当年数据是否为空，若空则触发手动 sync

### 4. 冲突检测

- 添加课位时检查同一老师同一时间段是否已有其他课位
- 检测到冲突 → 提示但不阻止（V1 不强制）

### 5. 多课位（同一学生多个时段）

- 一个学生可有多个 class_slots（如钢琴周六 9-10 + 乐理周日 14-15）
- 不同学生不冲突（除非同一老师同时段——见 #4）
- "今日课程"按时间排序展示全部

### 6. 余额下限

- 允许 negative balance（按 enrollment 计，如老师忘了记录付款）
- UI 红色显示（"余额：-2"），但不阻止扣减
- 提醒触发不受 negative 影响（按 `exam_levels.classes_count × 0.2` 阈值）

### 7. 时间约定

- **数据库**：`SET TIME ZONE 'UTC'`，所有 TIMESTAMPTZ 按 UTC 存查（Postgres 内部总是 UTC，session TZ 只影响显示）
- **应用层**：负责 UTC → Asia/Shanghai 转换（用于 display 和业务逻辑判断）
- **业务语义**：所有"今天 9 点上课"类陈述都是北京时间（UTC+8），应用层在写入/读出时做转换
- **V1 单时区假设**：中国大陆老师用户，UTC+8 始终是目标时区
- **未来多时区**：扩展为用户级 timezone 字段，应用层按用户 TZ 转换

### 8. 离线行为（V1 支持）

**写离线队列数据结构**（必须先定义，实施）：
```typescript
type QueuedOperation = {
  op_id: string;          // UUID v4，客户端生成，写入对应表的 client_op_id 字段
  op_type: 'attendance' | 'payment' | 'enrollment_update' | 'class_slot_update';
  payload: Record<string, unknown>;  // mutation 完整入参
  created_at: number;      // timestamp
  status: 'pending' | 'in_flight' | 'failed';
  retry_count: number;
  last_error: string | null;
};
```
存储在 IndexedDB store `pending_ops`。

**读离线**：
- TanStack Query 持久化缓存到 IndexedDB（via `@tanstack/query-async-storage-persister`）
- 应用启动时先加载缓存 → 立即可显示学生列表 / 今日课程等
- 网络可用时后台重新 fetch 更新

**写离线**：
- 用户操作（标记考勤、添加付款等）→ service 层先检查 `navigator.onLine`
  - 在线：直接调 RPC（带 `op_id` = `client_op_id`）
  - 离线：写入 IndexedDB 队列 + UI 乐观更新 + 标记 pending
- online 事件触发 / 应用启动时检测到 online → 遍历队列 → replay
  - 每个 op 调用对应 RPC，传 `op_id`（= `client_op_id`）实现幂等
  - 成功 → 标记 completed，删队列记录
  - 失败 → `retry_count + 1`，超过 5 次标记 failed，进入人工干预

**幂等性**：所有写操作通过 RPC 调用，RPC 接受 `client_op_id` 参数。对应表 `client_op_id UNIQUE` 约束保证：
- 双击 → 第二次 RPC 命中 UNIQUE 冲突，返回原记录，UI 无副作用
- 重连 replay → 同 `op_id` 已存在，跳过
- 客户端崩溃后重发 → 同样幂等

**冲突策略**：V1 last-write-wins（自用接受极小风险）

**UI 提示**：
- 顶部横幅：`离线模式 · X 个操作待同步`
- online 事件触发时自动隐藏
- failed 操作：横幅显示"X 个操作失败，请检查"

**在线检测**：
- `navigator.onLine` 浏览器事件
- 双保险：每 30s ping Supabase health endpoint（`/rest/v1/`），真实判断
- 检测到 online → 立即 replay 队列

**Tauri 特殊性**：
- Tauri WebView 支持 `localStorage` 和 IndexedDB
- localStorage ~5MB 上限；IndexedDB 配额更大（一般 50%+ 磁盘）
- 写队列用 IndexedDB store `pending_ops`

**限制**：
- 离线时新用户注册/邀请不能用（Supabase Auth 必须联网）
- 已登录用户 session 过期时离线 → 提示"请联网后重新登录"
- Reminder 仍依赖服务端 cron（离线不影响应用功能，但当天提醒可能看不到）

### 9. "今天"边界

- 应用层按 Asia/Shanghai 时区计算"今天"（用 `new Date().toLocaleDateString('zh-CN', { timeZone: 'Asia/Shanghai' })` 或类似）
- 应用启动时确定"今天"，**不自动跨日刷新**
- 重新打开应用时刷新"今天"
- 数据库查询"今天的课"：用 `WHERE scheduled_date = (CURRENT_TIMESTAMP AT TIME ZONE 'Asia/Shanghai')::date`

## Reminder Rules

### 触发条件（按报名）

- enrollment status = 'active' 且 `classes_remaining <= LOW_BALANCE_THRESHOLD`
- `LOW_BALANCE_THRESHOLD` 默认 **2**（V1 硬编码，V1.1 配置化）
- negative balance 也触发提醒
- 每个报名独立检测：小王可能"钢琴 3 级余额 2 待续费"+"乐理 1 级余额 5 正常"

**V1 设计选择**：因为 price_cents 是单节课学费，"周期"概念由 teacher 在 payment 时指定（"这次付 10 节课"），threshold 不再按 exam_levels 的固定值计算，统一用教师级默认（V1=2，V1.1 配置化）。

### 去重

- `dedup_key = enrollment_id || '_' || type || '_' || DATE_TRUNC('day', NOW())`
- 每天每个报名每个类型最多提醒一次
- DB UNIQUE constraint (`enrollment_id, dedup_key`) 强制

### 通知渠道（V1）

- **站内横幅（实时计算）**：登录后"今日课程"页面渲染时直接查 `enrollments.classes_remaining <= 2` 显示横幅（不依赖 cron 写库，UI 立即可用）
- **reminders 表（审计日志）**：GitHub Actions daily cron 仍跑，检测到阈值时 INSERT 一条记录（保留历史 + 后续可接邮件/通知推送）
- **Tauri 原生通知**：V1.1 集成 `tauri-plugin-notification`（macOS 通知中心）

**双轨设计**：横幅走 enrollments 实时计算（UI 响应快，不阻塞启动），reminders 表走 cron（审计 + 未来扩展）。

### Cron 配置

- **GitHub Actions**（外部定时，不依赖应用启动）
- cron 表达式使用 UTC：`0 1 * * *`（= 每日 09:00 CST = 01:00 UTC）
- **任务**：检测 `classes_remaining < threshold` 的学生 → INSERT 一条 reminder（依赖 UNIQUE 去重）→ done（无邮件发送）
- 已知风险：GitHub Actions 国内访问偶尔不稳定 → V1 接受，V1.1 评估备用方案（Supabase pg_cron Pro / Vercel cron）

## Approaches Considered

### A. Backend + Frontend stack（已锁定 Supabase + React）

| 方案 | Pros | Cons | Verdict |
|------|------|------|---------|
| 单文件 HTML + LocalStorage | 极简 | 无跨设备、无法扩展 | ❌ |

### B. Client Form Factor（Tauri vs Web）

| 方案 | Pros | Cons | Verdict |
|------|------|------|---------|
| **Tauri 桌面客户端** | 无部署、原生体验、自用最爽 | Rust 工具链、单机为主 | ✅ **已选** |

| PWA（纯 Web） | 零安装 | 离线能力有限 | 不采用（Tauri mobile 已锁定） |

### C. Future Org Migration Scope（重要补充）

文档之前说"加 organization_id 字段即可扩展"过于简化。**真实迁移范围**（V2 时执行）：

1. 新增 `organizations` 表（id, name, plan, settings）
2. 新增 `users.organization_id` 字段（FK）
3. 新增 `organization_members` 表（user_id, organization_id, role: owner/admin/teacher）
4. 修改 6 张表的 RLS 策略：从 `teacher_id = auth.uid()` 改为基于 organization 的 join + role check
5. 迁移现有数据：将每个 teacher_id 数据归入一个默认 organization
6. 加角色权限检查：teacher 只能看自己教的，admin 能看全组织

预估工作量：1-2 周（不算 V1 范围）。

### Critical User Journey: 标记一次考勤（端到端）

```
[用户]                       [应用 / Tauri]              [Supabase]
 │                                │                           │
 │ 1. 打开应用                    │                           │
 ├──────────────────────────────►│                           │
 │                                │ 2. TanStack Query: 拉取今日 │
 │                                │    scheduled_classes     │
 │                                ├──────────────────────────►│
 │                                │◄──────────────────────────┤
 │                                │                           │
 │ 3. 显示今日课程列表           │                           │
 │◄──────────────────────────────┤                           │
 │                                │                           │
 │ 4. 点击 "✓ 出席" (小王 9:00) │                           │
 ├──────────────────────────────►│                           │
 │                                │ 5. mutation:              │
 │                                │    service.markAttendance(│
 │                                │      scheduled_class_id, │
 │                                │      student_id,         │
 │                                │      result: 'attended') │
 │                                ├──────────────────────────►│
 │                                │                           │
 │                                │                   6. INSERT INTO attendance
 │                                │                   7. RPC mark_attendance: 更新 scheduled_classes.status
 │                                │                      + UPDATE enrollments SET
 │                                │                        classes_remaining = classes_remaining - 1
 │                                │                      WHERE id = enrollment_id
 │                                │◄──────────────────────────┤
 │                                │                           │
 │ 8. TanStack Query 失效        │                           │
 │    - scheduled_classes 缓存   │                           │
 │    - students 余额缓存        │                           │
 │                                │                           │
 │ 9. UI 自动刷新：               │                           │
 │    - "今日课程" 标记为已上     │                           │
 │    - 学生列表余额 7→6         │                           │
 │    - toast: "已记录，小王       │                           │
 │      余额 -1，剩 6 节"        │                           │
 │◄──────────────────────────────┤                           │
 │                                │                           │
```

**关键点**：
- Service 层封装 mutation（业务逻辑一处查）
- TanStack Query 失效缓存保证 UI 一致性
- DB 操作在一个 transaction（attendance insert + balance update）
- 失败回滚：toast 提示"网络错误，重试"，不扣余额

### Edge Cases & Failure Modes

| 场景 | 行为 | 处理位置 |
|------|------|----------|
| 网络断开 | 连接错误页面，操作禁用 | Tauri 全局拦截 |
| 双击 "出席" | mutation in-flight 锁 + DB 触发器防重 | TanStack Query + DB |
| 余额扣到负数 | 允许，UI 红色显示 | 不阻止 |
| 删除学生 | CASCADE 删 class_slots；scheduled_classes 软删（status='cancelled'） | DB + Service |
| 课位时间改到过去 | UI 校验禁止 | 前端表单 |
| 多个学生同时段 | 允许（冲突检测仅老师维度） | 不阻止 |
| 节假日当天的课 | 照常生成，老师手动改 status='cancelled' | UI 操作 |
| 邮箱验证没收到 | 引导用户查垃圾邮件 | 引导文案 |
| 应用启动时 cron 失败 | 客户端 RPC 兜底 | 启动时调用 `ensure_scheduled_classes_for_weeks` |
| 余额 ≥ 阈值已发过提醒 | DB UNIQUE 去重 | `reminders.dedup_key` |
| Tauri 启动时 Supabase 不可达 | 进入离线模式，显示横幅 + 缓存数据 | TanStack Query 持久化兜底 |
| GitHub Actions cron 失败 | 下次客户端启动 RPC 兜底 | 双层保险 |
| macOS Gatekeeper 拦截 | 引导"右键 → 打开" | README + 首次启动引导 |
| Xcode CLI Tools 未装 | tauri build 报错 | Step 0 必装 |
| Rust 工具链未装 | tauri build 报错 | Step 0 必装 |
| Supabase RLS 策略错误 | 自家数据不可见 → 调试 RLS SQL | 测试时手动验证 |
| 跨日不刷新"今天" | 用户重新打开应用时刷新 | UX 限制（明示在 §"Business Rules #9"） |

### Performance Considerations (V1 自用规模)

| 指标 | 目标 | 实际预估 |
|------|------|----------|
| 应用启动 | < 2s | 1-2s（Tauri 启动 + Vite bundle 加载） |
| 今日课程列表加载 | < 200ms | ~50ms（30 个学生 × 7 天查询） |
| 考勤标记 + 余额扣减 | < 500ms | ~200ms（DB transaction + cache invalidation） |
| 提醒检测（cron） | < 10s | ~2s（GitHub Actions + DB INSERT） |
| 包体积 | < 30MB | ~15MB（Tauri 最小 + React 19 + Tailwind 4） |

**性能瓶颈不在前端**（30 个学生规模太小）。瓶颈可能在：
- Supabase 网络延迟（国内 2-5s）—— 接受
- GitHub Actions cron 偶发延迟（5-10 分钟）—— 可接受
